# Feishu Org Intelligence CLI 设计稿

## 1. 项目概述

### 1.1 项目名称
Feishu Org Intelligence CLI

### 1.2 一句话定义
一个基于飞书 CLI 的组织智能操作系统，用世界模型、DRI 机制和 AI 协调替代传统层级管理中的信息路由与推进成本。

### 1.3 比赛定位
本项目面向“跨团队项目推进”场景，目标不是做一个普通任务管理工具，而是做一个可以把目标、项目、任务、阻塞、责任和简报串成闭环的组织协调层原型。

---

## 2. 背景与问题

传统层级管理承担三件事：
1. 信息汇总
2. 决策传导
3. 协调执行

当组织规模扩大、项目跨团队协同时，会出现典型问题：
- 信息散落在文档、群聊、表格、任务和会议中
- 项目推进依赖中层反复同步
- 责任边界模糊，出现“这事谁在推”不清楚的情况
- 关键风险往往在升级前缺少统一视图
- 管理者大量精力耗在状态对齐，而不是判断和资源调配

飞书天然沉淀了组织运行数据，但缺少一个“组织智能层”将这些对象统一建模并编排执行。

---

## 3. 核心理念

### 3.1 从 Hierarchy 到 Intelligence
项目借鉴“From Hierarchy to Intelligence”的核心思想：
- 过去组织通过层级来路由信息和协调执行
- 未来组织可以通过世界模型和智能层来完成同样的工作

### 3.2 不是“AI 助手”，而是“组织智能层”
本项目不只是给个人加一个 AI 助手，而是给团队增加一层智能协调能力：
- 看清目标与执行是否对齐
- 自动发现阻塞和风险
- 推荐 DRI（Directly Responsible Individual）
- 自动生成项目简报和待决策事项
- 将结果回写到飞书对象中

---

## 4. 场景定义：跨团队项目推进

### 4.1 典型场景
以一个跨团队产品项目为例：
- 产品、设计、研发、运营共同推进一个目标
- 多个任务存在依赖关系
- 某些问题横跨多个团队，容易出现 owner 缺失
- 管理者需要快速知道：哪里卡住了、谁该负责、下周重点是什么

### 4.2 成功标准
第一版 MVP 要能做到：
1. 用结构化模型描述一个跨团队项目
2. 自动识别风险和阻塞
3. 给关键问题推荐 DRI
4. 自动生成一份项目周报/决策简报
5. 把结果同步到飞书文档或多维表格中

---

## 5. 产品架构

### 5.1 总体架构
项目采用三层架构：

#### 数据层：Bitable
作为组织世界模型的核心承载层，负责管理结构化对象与关系。

#### 智能层：CLI
负责扫描、推理、归纳、建议与执行编排。

#### 展示/执行层：Docs + Chat + Tasks
用于对外呈现结果、分发信息、触发执行动作。

### 5.2 为什么选择 Bitable 驱动
相比纯文档驱动或纯任务驱动：
- 更适合做“世界模型”
- 结构化数据更方便扫描和计算
- 更适合体现飞书 CLI 的编排能力
- 同时保留 Docs 作为展示层，兼顾比赛观感

---

## 6. 世界模型设计

第一版最少包含 6 类核心对象。

### 6.1 Goals
描述组织目标。

建议字段：
- Goal ID
- Goal Name
- Period
- Priority
- Owner
- Status
- Progress
- Linked Projects
- Key Risks

### 6.2 Projects
描述项目对象。

建议字段：
- Project ID
- Project Name
- Related Goal
- DRI
- Team
- Stage
- Health
- Due Date

### 6.3 Tasks
描述执行项。

建议字段：
- Task ID
- Task Name
- Project
- Assignee
- DRI
- Priority
- Status
- Dependency
- Due Date
- Last Update
- Blocked Flag

### 6.4 Risks
描述项目风险。

建议字段：
- Risk ID
- Title
- Related Goal/Project
- Severity
- Status
- Owner / DRI
- Age
- Suggested Action

### 6.5 DRIs
描述跨团队问题的直接负责人。

建议字段：
- Topic / Problem
- DRI
- Scope
- Start Date
- Deadline
- Current Status
- Next Action

### 6.6 Signals
描述来自组织运行痕迹的信号。

建议字段：
- Source Type
- Source Link
- Related Object
- Signal Type
- Summary
- Confidence
- Timestamp

## 7. 智能层能力设计

### 7.1 风险扫描（Risk Scan）
识别典型问题：
- 超期任务
- 无 owner / 无 DRI 的关键事项
- 依赖阻塞
- 高优先级目标推进不足
- 状态长时间未更新

输出：
- 风险列表
- 严重等级
- 建议处理动作

### 7.2 DRI 推荐（DRI Suggest）
当发现跨团队问题时，系统给出：
- 谁最适合负责
- 哪些团队需要协作
- 推荐的下一步动作

第一版可先用规则驱动：
- 按目标 owner、项目 owner、任务 assignee、团队归属做推荐
- 后续再加入更智能的信号权重与历史经验

### 7.3 自动简报（Brief Generation）
自动生成：
- 本周进展
- 关键风险
- 待决策事项
- 下周重点

输出形式：
- 飞书文档
- Markdown 报告

### 7.4 组织健康度（Org Health）
第一版定义几个简单但有展示力的指标：
- Goal Progress Coverage
- Blocked Critical Task Count
- DRI Coverage Rate
- Risk Aging
- Decision Pending Count

---

## 8. CLI 命令设计

第一版不追求命令多，而追求闭环完整。

### 8.1 初始化
`foi init`
- 初始化项目配置
- 创建或连接基础数据结构
- 写入示例数据模板

### 8.2 同步
`foi sync`
- 从文档、任务、更新信息同步到世界模型

### 8.3 风险扫描
`foi risk scan`
- 输出当前项目阻塞与风险清单

### 8.4 DRI 推荐
`foi dri suggest`
- 针对关键问题推荐直接负责人和协作方

### 8.5 周报生成
`foi brief weekly`
- 生成项目周报/管理简报

### 8.6 组织健康度
`foi org health`
- 展示项目推进健康度和关键指标

---

## 9. MVP Demo 流程

### Step 1：初始化样例组织项目
创建一个跨团队项目：
- 1 个目标
- 2 个子项目
- 多个任务
- 若干依赖关系

### Step 2：注入问题场景
模拟真实推进问题：
- 某任务超期
- 某关键事项没有 DRI
- 某项目依赖阻塞
- 某高优目标进度偏慢

### Step 3：执行风险扫描
运行 `foi risk scan`
输出：
- 高风险问题
- 影响范围
- 建议动作

### Step 4：执行 DRI 推荐
运行 `foi dri suggest`
输出：
- 建议 DRI
- 推荐协作对象
- 下一步推进动作

### Step 5：生成管理简报
运行 `foi brief weekly`
输出：
- 文档版项目周报
- 风险摘要
- 待决策事项
- 下周重点

### Step 6：查看组织健康度
运行 `foi org health`
输出：
- 阻塞数
- DRI 覆盖率
- 风险老化情况
- 项目整体健康度

---

## 10. 错误处理与边界

第一版需要明确几个边界：
- 数据不完整时，允许降级输出而不是报错终止
- 无法自动判断 DRI 时，输出候选人和理由，而不是强行指定
- 对飞书对象写入失败时，要保留本地 Markdown 输出
- 对“智能判断”与“规则判断”明确区分，避免过度承诺

---

## 11. 测试与验证

第一版测试重点：
1. 数据结构是否能覆盖样例项目
2. 风险扫描是否能识别预设问题
3. DRI 推荐规则是否可解释
4. 周报生成是否结构清晰
5. CLI 到文档/表格的输出链路是否稳定

验证方式：
- 准备 demo 样例数据
- 对照预设结果检查 CLI 输出
- 检查生成文档与表格是否符合预期

---

## 12. 版本范围

### 12.1 第一版做什么
- 跨团队项目推进场景
- 结构化世界模型
- 风险扫描
- DRI 推荐
- 周报简报生成
- 基础健康度指标

### 12.2 第一版不做什么
- 不做真正复杂的 Agent 自主协作
- 不做组织级全量知识图谱
- 不做高度复杂的权限系统
- 不做实时监听所有飞书事件的重型架构

保持 MVP 可演示、可解释、可落地。

---

## 13. 比赛表达重点

对外叙事建议统一为：

> 大多数团队今天只是给每个人配一个 AI 助手；
> 我们想做的是给组织增加一个“智能协调层”。
> 它不替代专业判断，但能替代大量层级管理中的信息路由、状态对齐和责任识别成本。

一句话卖点：

> Feishu Org Intelligence CLI：把跨团队推进，从“靠人催”升级成“靠模型协调”。
