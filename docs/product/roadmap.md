# Feishu Org Intelligence CLI → 应用化路线

## 1. 先说结论

本项目第一阶段选择 **CLI MVP**，不是因为应用模式不重要，而是因为：

> 在“组织智能”这个命题里，真正的核心不是聊天入口，而是世界模型、风险识别、责任分配和管理输出。

因此我们刻意把建设顺序拆成两层：

1. **先做智能内核**：世界模型 + 规则引擎 + 管理简报
2. **再做交互外壳**：飞书应用 / Bot / 卡片 / 自动触发

换句话说：

> CLI 是第一阶段的控制台，飞书应用是第二阶段的交互层。

---

## 2. 为什么第一版先做 CLI，而不是直接做飞书应用

### 2.1 比赛阶段最重要的是闭环，不是壳子
飞书应用模式很强，但它天然会带来额外复杂度：
- 应用配置与权限 scope
- 用户授权
- 事件订阅
- 回调处理
- 卡片交互
- 消息触达
- 服务部署与稳定性
- 状态同步与容错

如果在项目最开始就直接压上这些成本，容易导致：
- 技术复杂度先爆炸
- 核心命题反而还没立住
- 时间被消耗在基础设施而不是产品价值上

所以第一版优先追求的是：
- 世界模型成立
- CLI 能跑通
- 风险扫描能解释
- DRI 推荐有说服力
- Docs / Bitable 闭环能展示

### 2.2 “更智能”不等于“先有应用”
应用只是一个入口层。

真正决定项目是否智能的，是它是否具备：
- 结构化组织世界模型
- 风险识别能力
- DRI 建议能力
- 信号整合能力
- 周报 / Brief 自动生成能力
- 对组织节奏的理解

如果没有这些能力，就算做成飞书 Bot，也只是一个“会聊天的壳子”。

而 CLI MVP 的价值，在于先把智能内核建立起来。

### 2.3 CLI 更适合承载“控制台”定位
对于本项目来说，CLI 的角色并不是过渡方案，而是一个有意义的产品组件：
- 它适合表达“组织操作系统”的概念
- 它适合做风险扫描、批量同步、世界模型编排
- 它适合比赛中展示“系统级控制台”的气质

因此第一版 CLI 并不低级，反而很适合比赛和原型验证。

---

## 3. 本项目的双层架构理解

### 3.1 第一层：组织智能内核
这是项目真正的核心，负责：
- 目标 / 项目 / 任务 / 风险 / DRI / Signals 建模
- 风险扫描
- DRI 推荐
- 组织健康度计算
- 周报 / 管理简报生成

这层已经在当前 MVP 中初步成立。

### 3.2 第二层：飞书应用交互层
这层负责：
- 用户触发入口
- 群内消息交互
- 卡片式反馈
- 自动提醒 / 自动播报
- 定时推送
- 审批 / 确认 / 跟进动作

换句话说：
- **智能内核回答“应该怎么判断”**
- **应用交互层回答“如何让团队使用它”**

---

## 4. 产品路线图

## Phase 1：CLI MVP（当前阶段）
### 目标
证明“组织智能层”成立。

### 已完成 / 正在完成
- 世界模型 schema
- Bitable 基础表结构
- demo 数据写入
- risk scan
- dri suggest
- brief weekly
- org health
- Docs 管理简报输出

### 阶段价值
这一阶段证明：
- 组织协作问题可以被结构化建模
- 风险与责任关系可以被 CLI 识别和表达
- 输出结果可以落到飞书对象中

---

## Phase 2：应用化（推荐下一阶段）
### 目标
把 CLI 能力包装成更自然的飞书使用体验。

### 推荐能力
#### 1. Bot 触发入口
例如：
- `/org risk scan`
- `/org brief weekly`
- `/org dri suggest`

#### 2. 卡片输出
把 CLI 当前文本输出升级成：
- 风险卡片
- DRI 建议卡片
- 周报摘要卡片
- 管理关注事项卡片

#### 3. 群内播报
支持：
- 每周自动推送项目简报
- 出现 Critical 风险时自动提醒
- DRI 缺失时自动告警

#### 4. 一键写回动作
在卡片或消息中触发：
- 指定 DRI
- 写入 Docs
- 更新 Bitable 状态
- 创建待办任务

### 阶段价值
这一阶段证明：
- 项目不仅能跑，还能被团队“用起来”
- 组织智能层具备真实交互形态

---

## Phase 3：信号驱动与更强自动化
### 目标
让系统从“被动调用”升级成“主动协同”。

### 推荐能力
#### 1. Signals 自动汇聚
从以下来源自动收集信号：
- 文档更新
- 任务变更
- 群消息
- 会议纪要

#### 2. 风险自动刷新
系统根据新信号：
- 自动识别阻塞
- 自动更新风险状态
- 自动标记老化风险

#### 3. 自动生成管理视图
- 每日摘要
- 周报
- 风险看板
- 决策待办清单

#### 4. 自动升级机制
当问题长期无人推进时：
- 自动提醒 DRI
- 自动升级给更高责任人
- 自动生成 briefing

### 阶段价值
这一阶段证明：
- 项目从“智能工具”进化成“轻量组织操作系统”

---

## 5. 比赛中该怎么讲这条路线

建议统一这样表达：

> 我们没有一开始就把精力放在应用壳子上，
> 而是先把真正的组织智能内核做出来：世界模型、风险识别、责任机制和管理输出。
> 
> 在这个基础上，飞书应用模式会成为第二阶段的交互层，让团队更自然地调用这些能力。

这段话的价值在于：
- 显得你是有产品判断的
- 说明你不是不会做应用，而是在控制节奏
- 说明你知道“智能内核”和“交互层”是两回事

---

## 6. 对评委最有说服力的一句话

> 第一阶段，我们用 CLI 证明“组织智能内核”成立；
> 第二阶段，我们用飞书应用模式把它变成团队真正会用的组织协作产品。

---

## 7. 最终产品愿景

最终形态不是一个单纯的命令行工具，也不是一个只会回答问题的 Bot。

最终形态应该是：

> 一个运行在飞书里的组织智能系统：
> Bitable 承载世界模型，Docs 承载管理输出，应用 / Bot 承载交互入口，
> 智能层持续理解组织状态、识别风险、建议责任归属，并帮助团队更快推进关键事项。
